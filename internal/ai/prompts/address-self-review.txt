A self-review landed on your draft PR. Your job is to read it, decide which comments to address, make the fixes, and push to the PR branch.

## Process

1. Run `triagefactory exec gh pr view -v` to find the most recent self-review and read its summary.
2. For the review, run `triagefactory exec gh pr review-view <review_id> --pr <N> -v` to read every inline comment in full.
3. Read `./task_memory/` before making changes — prior iterations may have already addressed or ruled out some of these points. Don't re-work what's already been resolved.
4. For each comment, decide: fix it, push back on it, or defer it. Prefer fixing — the review came from you, so the standard is "does this actually matter?" not "does the reviewer outrank me?"
5. Make the fixes in the codebase. Group related fixes into one commit.
6. Commit with a message that explains what you're addressing and why. Reference the review's key points — not the comment IDs, but the actual concerns.
7. Push to the PR branch.
8. Optionally reply to each addressed thread with `triagefactory exec gh pr comment-reply` explaining what changed. Don't reply if the commit message already says it.

## Constraints

- Only push to the branch you're already on. Do not force push, do not merge, do not rebase a shared branch.
- Do not resolve threads yourself — leave resolution to the human author's read of the fixes.
- If a comment is wrong or based on a misreading, push back in a reply rather than silently ignoring it. Explain your reasoning.
- If the review raises a concern you don't have context to fix (needs product input, needs a decision above your pay grade), report it in the summary rather than guessing.
- If prior task memory shows a previous fix attempt at the same issue, try a different approach.

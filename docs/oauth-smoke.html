Step 1 · Create the GitHub OAuth app

3

github.com/settings/developers → New OAuth App

For local smoke against localhost:3000:

  • Application name: Triage Factory (local smoke) — whatever's fine
  • Homepage URL: http://localhost:3000
  • Authorization callback URL: http://localhost:3000/auth/v1/callback

The callback URL is GoTrue's callback (the reverse-proxied /auth/v1/* path), not TF's. GitHub redirects there with the auth code; GoTrue then exchanges + 302s back to TF's /api/auth/callback internally.

Click Register application, then Generate a new client secret. Copy both into .env:

GH_CLIENT_ID=Ov23li...
GH_CLIENT_SECRET=<the generated secret>

Note: if you already brought up the stack with placeholder values, the gotrue container has them cached. You need docker compose up -d --force-recreate gotrue (not start) to pick up the new env. The triagefactory service doesn't need recreation — it reads GH_* only via GoTrue.

Step 3 · Walk the OAuth flow

Open in a browser (or follow the redirects with curl -L if you like watching them in a terminal):

http://localhost:3000/api/auth/oauth/github?return_to=/

What happens at each hop:

browser →
TF initiates the flow
GET /api/auth/oauth/github?return_to=/
Server generates a PKCE code_verifier + CSRF state token, packs them into a short-lived HttpOnly cookie (tf_oauth_state) HMAC-SHA256-signed with TF_COOKIE_SECRET as the MAC key — integrity only, no confidentiality, so the cookie body is base64-encoded JSON the browser could read but not forge. Then 302s the browser to GoTrue's /auth/v1/authorize with the matching code_challenge.
↓ 302
browser →
GoTrue requests provider redirect
GET /auth/v1/authorize?provider=github&code_challenge=...&flow_type=pkce
Hits TF first because of the reverse proxy in internal/server/auth_handlers.go — /auth/v1/* proxies through to GoTrue at gotrue:9999. GoTrue records the PKCE challenge keyed by the state token, then 302s to GitHub.
↓ 302
browser →
GitHub OAuth consent page
https://github.com/login/oauth/authorize?client_id=Ov23li…&redirect_uri=http://localhost:3000/auth/v1/callback&state=…
You see GitHub's "Authorize Triage Factory" page. Click authorize. GitHub redirects back to GoTrue's callback with a one-shot ?code=….
↓ 302 (with ?code=...)
browser →
GoTrue exchanges code for tokens
GET /auth/v1/callback?code=…&state=…
GoTrue (via the TF reverse proxy) calls github.com/login/oauth/access_token server-side to swap the code for a GitHub access token, fetches the user's email + identity, mints an RS256 access token signed with the GoTrue keypair, and 302s back to TF's callback with the auth code GoTrue assigned.
↓ 302
browser →
TF completes the session
GET /api/auth/callback?state=…&code=…
Server reads the tf_oauth_state cookie, validates the CSRF state match, calls GoTrue's /token?grant_type=pkce with the code_verifier to get the access + refresh tokens, verifies the access token via the JWKS Verifier (internal/auth/verify), creates a session row in public.sessions (access + refresh encrypted with TF_SESSION_ENCRYPTION_KEY), sets the sid HttpOnly cookie, and 302s to return_to (here, /).
↓ 302
browser →
DONE landed on the SPA, authenticated
GET /
The browser now carries sid=<uuid> HttpOnly. Every subsequent API request includes it; the auth middleware looks up the session row, decrypts the access token, verifies it via the Verifier, and attaches the JWT claims (orgID, userID) to the request context.

Step 4 · Verify the session works

5

Authenticated /api/me

# Use the cookie that the browser captured. Easiest:
# Chrome DevTools → Application → Cookies → localhost:3000 → sid → value
SID=<paste here>

curl -s -b "sid=$SID" http://localhost:3000/api/me
{"id":"<uuid>","email":"you@github.email","github_username":"you"}

# Without the cookie, the middleware refuses:
curl -s http://localhost:3000/api/me
unauthenticated  (HTTP 401)

# Logout — revokes the session row server-side, clears the cookie
curl -s -b "sid=$SID" -X POST http://localhost:3000/api/auth/logout
# Now the same SID is dead:
curl -s -b "sid=$SID" http://localhost:3000/api/me
unauthenticated  (HTTP 401)

What lives where

Reference for debugging — when something's wrong, this map tells you which container owns the state.

ArtifactLives inSet byUsed for
tf_oauth_state cookie
HttpOnly, 10min		 browser TF, in /api/auth/oauth/github Carries PKCE code_verifier + CSRF state across the redirect bounce so the callback can validate origin. HMAC-SHA256 signed with TF_COOKIE_SECRET — integrity, not confidentiality.
sid cookie
HttpOnly, 30d		 browser TF, in /api/auth/callback Session UUID. Every authenticated request carries this; auth middleware swaps it for the JWT claims.
public.sessions row Postgres TF, in /api/auth/callback Server-side session record. Stores AES-GCM-encrypted access + refresh tokens, plus user_id and last_seen_at.
auth.users row Postgres GoTrue, on first signup/login Canonical user identity. Has the GitHub provider entry under auth.identities.
JWKS (/.well-known/jwks.json) GoTrue, port 9999 GoTrue, from GOTRUE_JWT_KEYS at boot Public RSA key TF's Verifier fetches to validate access-token signatures.
Refresh logic TF process internal/server auth middleware When access token is <5min from expiry, middleware calls GoTrue's /token?grant_type=refresh_token via singleflight.Group (de-dupes concurrent refresh attempts per session).

Failure modes & what to look for

SymptomLikely causeWhere to look
GitHub shows "redirect_uri_mismatch" Your OAuth app's callback URL doesn't exactly match ${TF_PUBLIC_URL}/auth/v1/callback github.com/settings/developers → your app → callback URL field. Must include scheme, host, port, and /auth/v1/callback path.
Land on TF callback but get 500 State cookie missing or didn't survive the redirect chain DevTools → Network → look at /api/auth/callback request headers. Cookie: tf_oauth_state=... should be there. If missing, third-party cookie blocking is the usual culprit on Safari.
Land on TF callback, see "invalid state" CSRF state in cookie doesn't match the one returned by GoTrue Same as above — usually a cookie loss. Check TF logs: docker compose logs triagefactory | grep auth.
Land on / but /api/me returns 401 sid cookie not set OR session row not created DevTools cookies — confirm sid is present and HttpOnly. Then docker compose exec postgres psql -U supabase_admin -c "SELECT id, user_id, created_at FROM public.sessions ORDER BY created_at DESC LIMIT 5" to see if the row landed.
Verifier errors in TF logs JWKS fetch failed or signature mismatch (kid drift after rotate) curl -s http://localhost:9999/.well-known/jwks.json | jq — verify the kid matches what GoTrue is signing with (decode the access token via jwt.io; the kid in the header should match).
Browser silently lands on / unauthenticated The 302 chain succeeded but the callback handler bailed before setting sid TF logs at request time. Look for log lines from internal/server/auth_handlers.go:handleOAuthCallback.

Appendix · The two compose fixes

These are the patches surfaced during the Tier 1 + 2 smoke. Apply before bringing the stack up; otherwise the auth flow can't even begin.

1. Escape $ in the postgres-postinit heredoc

  postgres-postinit:
    command:
      - |
        set -eu
        psql -h postgres -U supabase_admin -d postgres -v ON_ERROR_STOP=1 <<-'SQL'
-       \set auth_admin_password `echo "$SUPABASE_AUTH_ADMIN_PASSWORD"`
-       \set authenticator_password `echo "$AUTHENTICATOR_PASSWORD"`
+       \set auth_admin_password `echo "$$SUPABASE_AUTH_ADMIN_PASSWORD"`
+       \set authenticator_password `echo "$$AUTHENTICATOR_PASSWORD"`
        ALTER USER supabase_auth_admin WITH LOGIN PASSWORD :'auth_admin_password';
        ALTER USER authenticator WITH LOGIN PASSWORD :'authenticator_password';
        SQL

Without $$, compose substitutes $AUTHENTICATOR_PASSWORD from the host env at parse time. Host env doesn't have that name (we only set TF_AUTHENTICATOR_PASSWORD), so it interpolates to empty string and the ALTER ROLE sets an empty password.

2. Connect as supabase_admin for TF_DATABASE_URL

  triagefactory:
    environment:
-     TF_DATABASE_URL: postgres://postgres:${POSTGRES_PASSWORD:?...}@postgres:5432/postgres
+     TF_DATABASE_URL: postgres://supabase_admin:${POSTGRES_PASSWORD:?...}@postgres:5432/postgres

The supabase/postgres image demotes postgres to non-superuser and supautils blocks reserved-role ALTERs from it. The migration baseline uses ALTER DEFAULT PRIVILEGES FOR ROLE supabase_admin which requires supabase_admin membership. Per internal/db/postgres/store.go:20-26, TF_DATABASE_URL is the admin pool DSN by design and should always be supabase_admin. Same password (POSTGRES_PASSWORD is shared by the supabase image's initdb convention).
